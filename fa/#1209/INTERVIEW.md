# 인터뷰 기록 — #1209 투자제안 알림톡 상품변경 지원

> 담당자와의 인터뷰를 통해 SPEC을 구체화한 과정을 기록.
> 이후 유사 feature 작업 시 참고용.

---

## 배경 및 초기 요구사항

**요청 내용**

FA가 미결계약 또는 기계약 상태인 고객에게 상품변경을 위한 포트폴리오 제안 알림톡을 발송할 수 있어야 한다.

**현재 상태**

- 투자제안 알림톡은 신규 고객(계약 없음) 가정으로만 동작 중
- `sendInvestStrategy({ id, portfolioCode, accountType })` — 계약 정보 없음
- 계약현황 탭(`PortfolioContractTab`): 기계약만 표시, 미결계약 UI 없음
- 미결계약 관련 API 없음

---

## Q&A 기록

### [UI 방향] 기획안에 대한 의견

**Q.** 기획자는 계약현황에 미결계약 섹션을 따로 만들고 각 항목 클릭 시 사이드모달로 발송하는 안을 제안했는데, 신규/미결/기계약 각각 따로 모달이 생기는 게 적절한지?

**A.** 중복 동작 모달이 많아지는 건 좋지 않다. 투자제안 화면 따로, 계약 확인 화면 따로로 명시적으로 구분되면 좋겠다.

---

### [UI 방향] 화면 구성 선택

**Q.** 3가지 UI 프로토타입 중 선택  
- A: 2탭 재편 (투자제안 통합 + 계약현황 조회 전용)  
- B: 계약현황 탭의 각 계약 카드에 상품변경 버튼 추가  
- C: 계좌타입 중심 단일 뷰

**A.** **B안 선택** — 계약현황 탭의 미결/기계약 카드에 상품변경 제안 버튼 추가. 버튼 클릭 시 별도 사이드모달(기획안 방식) 허용.

---

### [범위] 포트폴리오 실행 탭

**Q.** 현재 포트폴리오 실행 탭(KB증권 전용 발송)을 이번 feature에서 어떻게 처리할지?

**A.** 그대로 유지. 이번 scope 제외.

---

### [범위] 상품변경 발송 가능 상태

**Q.** 상품변경 알림톡 발송이 가능한 계약 상태는?

**A.** 미결계약(투자제안 발송 후 고객이 앱에 진입해 미결계약이 생성된 상태, 포트폴리오 선택 완료 여부 무관) + 기계약(계약 체결 완료 상태).

---

### [API] 상품변경 발송 API 처리 방식

**Q.** 기존 API에 파라미터 추가 vs 새 endpoint?

**A.** 기존 `sendInvestStrategy`에 파라미터 추가.

---

### [범위] 미결계약 UI

**Q.** 미결계약 카드 UI를 새로 만드는 것도 #1209 scope인가?

**A.** 맞음. 미결계약 카드 자체를 새로 만든다. 미결계약 섹션/기계약 섹션을 나눌지는 조정이 필요하나, B안 기준(계약현황 탭 내 통합 표시)으로 진행.

---

### [발송 후] 포트폴리오 제안 탭 업데이트

**Q.** 상품변경 발송 후 포트폴리오 제안 탭 발송 정보도 업데이트 필요한가?

**A.** 필요함. 발송 성공 후 포트폴리오 제안 탭의 발송 이력 API 재조회.

---

### [모달] 기존 운용 중 MP 표시

**Q.** 상품변경 모달에서 기존 운용 중 포트폴리오를 보여줄 건가?

**A.** 기계약의 운용 중 MP는 모달에서 보여주면 좋다. 미결계약에서 발송했던 제안 MP도 동일하게 표시한다. 두 케이스 모두 현재 포트폴리오와 동일한 MP 재선택 시 발송 버튼을 비활성화한다.

---

### [미결계약 카드] D-N 만료일 표시

**Q.** 미결계약 카드에 D-N(만료까지 남은 일수) 표시를 원하는가? PCT 생성일 데이터 포함 여부?

**A.** D-N 표시 원함. 단, 백엔드에서 미결계약 삭제 정책이 아직 미적용 상태 — 현재는 미결 생성일만 제공 예정. 삭제 예정일은 이후 작업 시 추가 제공될 예정. 프론트에서 임의 계산하여 표시하는 방식은 비권장.

> **결론**: 현재는 미결 생성일 표시. D-N은 백엔드에서 삭제 예정일 필드 제공 이후 추가 구현.

---

### [발송 후] 계약현황 탭 refresh

**Q.** 발송 성공 후 계약현황 탭도 데이터를 다시 불러와야 하나?

**A.** 불필요. 발송 후 바로 미결계약이 생성되는 구조가 아님 — 고객이 알림톡으로 앱에 진입해야 미결계약이 생성됨. 계약현황 탭은 페이지 재진입 시 자연스럽게 리패치되면 충분. 포트폴리오 제안 탭 API만 발송 후 업데이트하면 됨.

---

### [구조 변경] 탭 구조 변경 배경 (2차 인터뷰)

**Q.** 초기 인터뷰에서 3탭 유지로 결론났는데, 3탭 → 2탭 + 별도 계약관리 페이지로 바뀐 이유는?

**A.** 원래 탭 순서가 계약 흐름(투자제안 → 실행 → 현황)을 따랐으나, 요구사항이 "한 화면에서 모든 상태를 파악하고 관리"로 바뀌면서 기능을 한 곳으로 몰았다. 그 과정에서 포트폴리오 실행 탭의 실행 기능이 빠져 탭 이름·역할이 변경됐다. 계약 현황은 내용이 무거워져 3depth 탭 안에 두는 것이 적합하지 않다고 판단하여 별도 페이지로 분리하기로 결정했다.

---

### [메뉴] 계약관리 메뉴 위치 (2차 인터뷰)

**Q.** 투자설계 메뉴 어디에 계약관리가 위치하는가?

**A.** 투자설계 메뉴 가장 아랫부분. 투자제안을 한 다음 계약은 순서상 마지막에 생기는 결과물이므로 하위에 배치.

---

### [채널] 타 증권사에 계약관리 버튼이 없는 이유 (2차 인터뷰)

**Q.** 계약 채널 탭에서 KB증권 카드에만 "계약관리" 버튼이 있는 이유는?

**A.** 타 증권사(한국투자증권, 신한투자증권)는 우리 서비스 내에서 계약이 불가하다. FA 플랫폼에서 계약 관리가 가능한 증권사는 KB증권뿐이다.

---

### [발송 이력] 최근 제안 API 출처 (2차 인터뷰)

**Q.** 계약관리 페이지의 최근 제안 데이터는 어느 API에서 오는가?

**A.** 포트폴리오 제안 탭의 발송 이력 API 재사용 예정. 백엔드 작업 상황에 따라 변경될 수 있음.

---

### [발송 이력] 최근 제안 표시 단위 (2차 인터뷰)

**Q.** 최근 제안이 계좌유형 그룹 단위인지 카드 단위인지?

**A.** 카드별로 따로. 신규 발송한 건은 미결계약 카드가 생성될 때 그 카드 하단에 붙는다.

---

### [발송 후] 갱신 대상 API (2차 인터뷰)

**Q.** 발송 성공 후 정확히 어떤 API를 재조회해야 하는가?

**A.** 계약관리 페이지의 발송 이력 API만 갱신. 포트폴리오 제안 탭이 없어지므로 해당 탭 갱신은 불필요.

---

### [사이드모달] 신규 제안 사이드모달 구성 (2차 인터뷰)

**Q.** 신규 제안 버튼 클릭 시에도 상품변경과 동일한 사이드모달을 사용하는가?

**A.** 동일한 사이드모달 사용. 현재 포트폴리오 표시 영역은 조건에 따라 표출/숨김 처리한다 (신규 제안 시 숨김, 상품변경 시 표출).

---

### [계약 상태] 해지 구분 기준 (2차 인터뷰)

**Q.** 해지완료(카드 미노출)와 해지 진행 중(카드 표시·상품변경 버튼 미노출)의 상태코드 기준은?

**A.** `policy/bw-is/계약상태.md` 기준. KBS 자문:
- 해지 진행 중 → 카드 표시, 상품변경 버튼 미노출: `2231`(현물해지 처리대기), `2234`(현물해지 출금대기)
- 해지완료 → 카드 미노출: `2290`

---

### [구조 변경] 탭 구조 재변경 — 3탭으로 복원 (3차 인터뷰)

**Q.** 2차 인터뷰에서 2탭 + 계약관리 별도 페이지로 결정했는데, 다시 3탭(포트폴리오 제안 / 계약 채널 / 계약 관리)으로 변경한 배경은?

**A.** 계약관리를 별도 메뉴로 분리하면 FA 입장에서 투자제안과 계약관리를 오가야 하는 불편이 있다. 투자제안 페이지의 3번째 탭으로 넣어 한 화면 안에서 제안→채널→관리 흐름을 유지하기로 결정.

---

### [발송 이력] 최근 발송 표시 단위 재변경 — 계좌유형당 1건 (3차 인터뷰)

**Q.** 2차 인터뷰에서 카드별 1건이었는데, 계좌유형당 1건으로 변경한 이유는?

**A.** 카드마다 이력을 붙이면 정보가 과도해진다. 계좌유형 그룹 타이틀 바로 아래에 가장 최근 1건만 표시하는 것으로 충분. 신규 제안과 상품변경을 구분하지 않고 해당 계좌유형의 가장 최근 발송 1건을 표시한다.

---

### [기계약 카드] 필드 전체 항상 노출 (3차 인터뷰)

**Q.** 기계약 카드에 18개 필드를 모두 펼쳐서 표시하면 카드가 길어진다. 접기/더보기 처리 필요 여부는?

**A.** 전체 항상 노출. 접기 처리 불필요.

---

### [기계약 카드] 다중계약 시 정렬 순서 (3차 인터뷰)

**Q.** 연금저축·종합위탁에서 기계약 카드가 2개 이상일 때 정렬 순서는?

**A.** 아직 정해진 정책이 없다. 필요한 부분이므로 정책 확정 후 반영 예정.

> **결론**: 정책 미확정. SPEC에 미확정으로 명시.

---

### [발송 후] 상품변경 발송 후 계약 데이터 재조회 (3차 인터뷰)

**Q.** 상품변경 발송 성공 시 미결계약 카드의 포트폴리오명이 stale해질 수 있다. 계약 데이터를 재조회해야 하나?

**A.** 재조회 안 함. 발송 이력만 갱신. 계약 데이터는 다음 페이지 진입 시 자연스럽게 갱신.

---

### [표시 정책] 오버뷰 vs 계약 관리 탭 빈 그룹 차이 (3차 인터뷰)

**Q.** 계약 관리 탭은 빈 그룹도 항상 표시하고 오버뷰는 미표시인데, 이 차이가 의도적인가?

**A.** 의도적이다. 계약 관리 탭은 신규 제안 진입점 역할이므로 항상 표시. 오버뷰는 요약 목적이므로 데이터 없으면 숨김.

---

### [미결계약 카드] 상품변경 버튼 진행단계별 조건 (3차 인터뷰)

**Q.** 미결계약의 진행단계(contractState)에 따라 상품변경 버튼을 숨겨야 하는 경우가 있는가?

**A.** 항상 노출. 발송 가능 여부는 백엔드에서 차단.

---

### [계약 채널] 외부 증권사 카드 동작 (3차 인터뷰)

**Q.** 한국투자증권·신한투자증권 카드의 "계약 방법" 버튼 동작은 이번 scope에서 변경이 있는가?

**A.** 기존 동작 유지. scope 외.

---

### [오버뷰] 개별 행 클릭 동작 (3차 인터뷰)

**Q.** 오버뷰에서 개별 계약 행 클릭 시 상세 이동 또는 계약 관리 탭 이동 등의 동작을 추가할 계획이 있는가?

**A.** 동작 없음 유지.

---

### [사이드모달] 포트폴리오 목록 필터링 (3차 인터뷰)

**Q.** 사이드모달에서 표시되는 포트폴리오 목록은 해당 계좌유형에 맞는 MP만 필터링하는가?

**A.** 계좌유형별 필터링. IRP 그룹에서 열면 IRP용 MP만 표시.

---

### [발송] 실패 후 재시도 제한 (3차 인터뷰)

**Q.** 발송 실패 후 사이드모달에서 재시도에 횟수 제한이나 쿨다운이 있는가?

**A.** 제한 없음. 사이드모달 유지 상태에서 자유롭게 재시도 가능.

---

### [미결계약] 만료/삭제 시 화면 반영 (3차 인터뷰)

**Q.** 미결계약이 백엔드에서 삭제된 경우 FA 화면에서 실시간 반영이 필요한가?

**A.** 실시간 반영 불필요. 다음 페이지 진입 시 자연스럽게 반영.

---

## 결론 — SPEC 반영 사항

| 항목 | 결정 내용 |
